# Requirements Document: 候補投稿フォーム機能

## Overview

このドキュメントは、候補投稿フォーム機能の要件を定義します。この機能は、認証済みユーザーが対局の次の一手候補を投稿するためのWebフォームを提供します。

## Functional Requirements

### 1. ページアクセスとルーティング

**1.1** システムは `/games/[gameId]/candidates/new` パスで候補投稿ページを提供しなければならない

**1.2** システムはクエリパラメータ `turnNumber` でターン番号を指定可能にしなければならない（オプション）

**1.3** `turnNumber` が指定されない場合、システムは現在のターン番号を使用しなければならない

**1.4** システムは存在しないゲームIDに対して 404 ページを表示しなければならない

### 2. 認証と認可

**2.1** システムは認証済みユーザーのみが候補を投稿できるようにしなければならない

**2.2** システムは未認証ユーザーに対して「認証が必要です」というエラーメッセージを表示しなければならない

**2.3** システムはアクセストークンを自動的に取得し、API リクエストに含めなければならない

### 3. 位置選択機能

**3.1** システムはインタラクティブな盤面を表示し、ユーザーがセルをクリックして位置を選択できるようにしなければならない

**3.2** システムは選択されたセルをハイライト表示しなければならない

**3.3** システムは位置が選択されていない場合、「位置を選択してください」というエラーメッセージを表示しなければならない

**3.4** システムは送信中はセル選択を無効化しなければならない

**3.5** システムは選択された位置を "row,col" 形式（例: "2,3"）で保存しなければならない

### 4. プレビュー機能

**4.1** システムは位置が選択された場合、その位置に石を置いた後の盤面をプレビュー表示しなければならない

**4.2** システムはプレビューを現在のプレイヤー（黒または白）に基づいて計算しなければならない

**4.3** システムは位置が選択されていない場合、プレビューを表示しないようにしなければならない

### 5. 説明文入力機能

**5.1** システムは説明文を入力するためのテキストエリアを提供しなければならない

**5.2** システムは説明文の最大文字数を 200 文字に制限しなければならない

**5.3** システムは現在の文字数を「X/200文字」の形式で表示しなければならない

**5.4** システムは説明文が空の場合、「説明文を入力してください」というエラーメッセージを表示しなければならない

**5.5** システムは説明文が 200 文字を超えた場合、「説明文は200文字以内で入力してください」というエラーメッセージを表示しなければならない

**5.6** システムはリアルタイムで文字数をカウントし、200 文字超過時に即座にエラーを表示しなければならない

### 6. フォーム送信機能

**6.1** システムは「候補を投稿」ボタンを提供しなければならない

**6.2** システムは送信中はボタンを無効化し、「送信中...」と表示しなければならない

**6.3** システムは送信前にクライアント側でバリデーションを実行しなければならない

**6.4** システムはバリデーションエラーがある場合、API 呼び出しを実行せずにエラーメッセージを表示しなければならない

**6.5** システムは成功時に候補一覧ページ (`/games/[gameId]`) へリダイレクトしなければならない

### 7. キャンセル機能

**7.1** システムは「キャンセル」ボタンを提供しなければならない

**7.2** システムはキャンセルボタンクリック時に前のページに戻らなければならない

### 8. エラーハンドリング

**8.1** システムは API エラーに対して適切なエラーメッセージを表示しなければならない

**8.2** システムは 401 エラーに対して「認証が必要です。ログインしてください。」と表示しなければならない

**8.3** システムは 409 エラーに対して「この位置の候補は既に存在します。別の位置を選択してください。」と表示しなければならない

**8.4** システムは 400 INVALID_MOVE エラーに対して「この位置には石を置けません。別の位置を選択してください。」と表示しなければならない

**8.5** システムは 400 VOTING_CLOSED エラーに対して「投票期間が終了しています。」と表示しなければならない

**8.6** システムはその他のエラーに対して「候補の投稿に失敗しました。もう一度お試しください。」と表示しなければならない

**8.7** システムはエラー発生後、フォームを再送信可能な状態に戻さなければならない

### 9. API 統合

**9.1** システムは POST /api/games/:gameId/turns/:turnNumber/candidates エンドポイントを呼び出さなければならない

**9.2** システムはリクエストボディに `position` と `description` を含めなければならない

**9.3** システムは Authorization ヘッダーに Bearer トークンを含めなければならない

**9.4** システムは 201 Created レスポンスを成功として扱わなければならない

## Non-Functional Requirements

### 10. パフォーマンス

**10.1** システムは初期ページロードを 3 秒以内に完了しなければならない

**10.2** システムはフォーム送信を 5 秒以内に完了しなければならない

**10.3** システムはプレビュー計算を 500ms 以内に完了しなければならない

### 11. アクセシビリティ

**11.1** システムはすべてのフォーム要素に適切な label を提供しなければならない

**11.2** システムはエラーメッセージをフォーム要素と関連付けなければならない

**11.3** システムはキーボードのみで操作可能でなければならない

**11.4** システムはスクリーンリーダーで読み上げ可能でなければならない

**11.5** システムは適切な ARIA 属性を使用しなければならない

### 12. レスポンシブデザイン

**12.1** システムはモバイル（320px 以上）、タブレット（768px 以上）、デスクトップ（1024px 以上）で正しく表示されなければならない

**12.2** システムはモバイルでタッチ操作に最適化されなければならない

**12.3** システムは盤面のセルサイズをデバイスに応じて調整しなければならない

### 13. セキュリティ

**13.1** システムは XSS 攻撃を防ぐために説明文を適切にエスケープしなければならない

**13.2** システムは CSRF 攻撃を防ぐために適切な対策を実装しなければならない

**13.3** システムはクライアント側とサーバー側の両方でバリデーションを実行しなければならない

### 14. ユーザビリティ

**14.1** システムはフォームの各セクションに明確なラベルを提供しなければならない

**14.2** システムはエラーメッセージを赤色で目立つように表示しなければならない

**14.3** システムは送信中の状態を視覚的に明確に示さなければならない

**14.4** システムはプレースホルダーテキストで入力例を提供しなければならない

### 15. 互換性

**15.1** システムは最新の Chrome、Firefox、Safari、Edge で動作しなければならない

**15.2** システムは iOS Safari と Android Chrome で動作しなければならない

## Data Requirements

### 16. 入力データ

**16.1** システムは位置データを "row,col" 形式で受け付けなければならない

**16.2** システムは row と col が 0〜7 の整数であることを検証しなければならない

**16.3** システムは説明文を 1〜200 文字の文字列として受け付けなければならない

### 17. 出力データ

**17.1** システムは API レスポンスから candidateId を取得しなければならない

**17.2** システムは成功時にリダイレクト先 URL を構築しなければならない

## Integration Requirements

### 18. 既存コンポーネントとの統合

**18.1** システムは既存の Board コンポーネント（spec 15）を再利用しなければならない

**18.2** システムは既存の BoardPreview コンポーネント（spec 23）を再利用しなければならない

**18.3** システムは既存の API Client（candidates.ts）に `createCandidate` 関数を追加しなければならない

**18.4** システムは既存の ApiError クラスを使用してエラーハンドリングを実装しなければならない

**18.5** システムは既存の storageService を使用してアクセストークンを取得しなければならない

### 19. API 仕様との整合性

**19.1** システムは spec 19（POST 候補投稿 API）の仕様に準拠しなければならない

**19.2** システムは API のエラーコード（VALIDATION_ERROR, INVALID_MOVE, VOTING_CLOSED, CONFLICT）を正しく処理しなければならない

## Testing Requirements

### 20. ユニットテスト

**20.1** システムは CandidateForm コンポーネントのユニットテストを提供しなければならない

**20.2** システムは API Client の `createCandidate` 関数のユニットテストを提供しなければならない

**20.3** システムはバリデーションロジックのユニットテストを提供しなければならない

### 21. プロパティベーステスト

**21.1** システムは説明文の長さ制約に対するプロパティテストを提供しなければならない

**21.2** システムはリアルタイムバリデーションに対するプロパティテストを提供しなければならない

### 22. 統合テスト

**22.1** システムは完全なフォーム送信フローの統合テストを提供しなければならない

**22.2** システムはエラーケースの統合テストを提供しなければならない

### 23. E2E テスト

**23.1** システムはユーザーが候補を投稿できることを検証する E2E テストを提供しなければならない

**23.2** システムはバリデーションエラーが表示されることを検証する E2E テストを提供しなければならない

## Constraints

### 24. 技術的制約

**24.1** システムは Next.js 16 App Router を使用しなければならない

**24.2** システムは React 19 を使用しなければならない

**24.3** システムは TypeScript strict mode を使用しなければならない

**24.4** システムは Tailwind CSS でスタイリングしなければならない

**24.5** システムは Zod でバリデーションスキーマを定義しなければならない

### 25. ビジネス制約

**25.1** システムは MVP ではオセロ（リバーシ）のみをサポートしなければならない

**25.2** システムは 1 ユーザーあたり 1 ターンに複数の候補を投稿可能でなければならない

**25.3** システムは投票締切後の候補投稿を拒否しなければならない

## Acceptance Criteria

### 26. 機能の完成条件

**26.1** ユーザーが盤面上でセルをクリックして位置を選択できる

**26.2** 選択されたセルがハイライト表示される

**26.3** 選択された位置のプレビューが表示される

**26.4** 説明文を入力でき、文字数がリアルタイムで表示される

**26.5** バリデーションエラーが適切に表示される

**26.6** フォーム送信が成功し、候補一覧ページへリダイレクトされる

**26.7** API エラーが適切に処理され、エラーメッセージが表示される

**26.8** すべてのテストが合格する

**26.9** アクセシビリティチェックが合格する

**26.10** レスポンシブデザインがすべてのデバイスで正しく動作する
